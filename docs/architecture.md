# Scalp pipeline architecture

This doc describes the scalp trading system after the **strip-down**
(branch lineage: `strip/scalp-simplify`). The system was previously a stack of
parallel layers (`v2` core, `v3` temporal filter, `v4` regime classifier,
`v5` regime-evidence gate). Those layers have been removed. What remains is a
single, linear pipeline:

```
candles → session_structure_composer_v1 grid
        → A/B/C staged backtest + pooled edge-significance promote
        → live execute (minimal gates)
```

All code lives under [`lib/scalp/`](../lib/scalp/); the composer pipeline is in
[`lib/scalp/composer/`](../lib/scalp/composer/).

## The composer

`session_structure_composer_v1` is the only production strategy. Candidates are
generated by a **block-mixing grid**
([`buildScalpComposerSessionStructureComposerGrid`](../lib/scalp/composer/sessionStructureComposer.ts)):
every compatible combination of `context × level × trigger × confirmation ×
management` session-structure blocks becomes a candidate tune. Generation is
steered by adaptive priors and a novelty budget (exploit / adjacent / explore
lanes) so the search reaches good cells sooner. Each candidate carries a
deterministic `tuneId`; the execution plan is resolved from the tuneId/blocks
via the still-live machinery in
[`composerExecution.ts`](../lib/scalp/composer/composerExecution.ts).

## Candidate lifecycle

```
discover / research    (composer research job — local bulk or manual trigger)
   → block-mixing grid expansion → in-memory candidates
   → A/B/C staged replay backtest (increasing windows)
   → stage-C edge significance (lowerBoundR, pooled across the cross-symbol cohort)
   → write candidate + worker evidence (scalp_v2_candidates / scalp_v2_deployments)

promote                (every 15 min)
   → gates (see below); flips enabled = TRUE on qualifying rows

live execute           (every 1 min)
   → composed entry gates → place / manage / close trades

reconcile              (every 2 min)
   → broker position reconciliation
```

Research/worker/discovery are run from the terminal via
[`scripts/research-local-bulk.ts`](../scripts/research-local-bulk.ts) (composer
mode) or by manually invoking the composer crons; they are intentionally not on
a vercel schedule (too long-running for the serverless timeout). An optional
**day-finalist robustness** pass (`dayRobustness`) can gate promotion of day
candidates and runs after a full local bulk drain.

## Promotion gates (composed in order)

Evaluated in [`runScalpComposerPromotePass`](../lib/scalp/composer/pipeline.ts):

1. Strategy must be `session_structure_composer_v1` (retired composers are flat-disabled — see below).
2. Worker stage-C pass.
3. Worker freshness (stage-C evidence computed for the current completed week).
4. Backtest 4-week rolling netR ≥ threshold.
5. Edge sample size: pooled stage-C trades ≥ promotion floor.
6. Edge significance: pooled `lowerBoundR` (meanR − 1.64·stderrR, net of fees) ≥ threshold.

There is no longer a holdout, drift, temporal-variant, prior-score, or
regime/cell gate. The lowerBoundR significance test (computed by
`computeScalpComposerV3EdgeScore`) and the forex news blackout
(`evaluateScalpComposerV3NewsBlackout`) are the only pieces retained from the
former v3 layer.

## Live entry gates (composed in order)

Each blocked entry is journaled with the reason codes from every gate that
fired. Existing positions are NOT affected — they reconcile and close
naturally; gates only block NEW entries.

| # | Gate | Reason codes |
|---|---|---|
| 1 | Promotion gate block list (`deployment.promotion_gate.entryBlockReasonCodes`) | varied (scope, freshness, …) |
| 2 | Forex news blackout | `V3_NEWS_BLACKOUT*` |
| 3 | Retired-composer block | `COMPOSER_RETIRED_NEW_ENTRIES_BLOCKED` |
| 4 | Sunday no-trade window | `SUNDAY_*` |

Beyond these, the engine itself enforces risk controls (kill switch,
daily-loss limit, session window, entry cooldowns).

## Retired composers

`model_guided_composer_v2` and `day_model_guided_composer_v1` are retired.
Their strategy implementations are no longer registered, but their id constants
survive (`MODEL_GUIDED_COMPOSER_V2_STRATEGY_ID`,
`DAY_MODEL_GUIDED_COMPOSER_V1_STRATEGY_ID`) so the live guard
`isRetiredComposerStrategyId` can flat-disable any lingering deployment: new
entries blocked, existing positions managed to close via cfg-driven exits, row
disabled once flat.

## Cron schedule (vercel.json)

| Schedule | Path | Purpose |
|---|---|---|
| `* * * * *` | `/api/scalp/composer/cron/execute` | Live execution loop |
| `*/2 * * * *` | `/api/scalp/composer/cron/reconcile` | Position reconciliation |
| `*/15 * * * *` | `/api/scalp/composer/cron/promote` | Auto-promote qualifying rows |
| `*/30 * * * *` | `/api/scalp/composer/cron/load-candles` | 1m candle ingestion (both venues) |

The dashboard is served at [`/legacy`](../pages/legacy.tsx).

## Storage layout

| Table | Notes |
|---|---|
| `scalp_v2_deployments` | Authoritative deployment registry; promotion gate + worker evidence in JSONB |
| `scalp_v2_candidates` | Research candidates + staged backtest evidence |
| `scalp_candle_history_weeks` | 1m candles, per-week blob |
| `scalp_v2_journal` | Execution event log; gate decisions land here |
| `scalp_v2_runtime_config` | Singleton runtime config (kill switch, scope, risk profile) |

Manually retired rows survive as DB rows with `candidate_id = NULL` and
`enabled = FALSE` — history is preserved, nothing is hard-deleted.

## Removed in the strip-down

For reference, these subsystems were removed: v3 temporal-variant / holdout /
bootstrap / drift / prior-score, the v4 regime classifier (`lib/scalp/regimes/`),
the v5 regime-evidence layer (`lib/scalp/research/`), the adaptive meta-selector
(`lib/scalp/adaptive/`), the surrogate stage-A pre-screen, and the evolutionary
offspring search. Their `SCALP_COMPOSER_SURROGATE_*`,
`SCALP_COMPOSER_SESSION_EVOLUTION_*`, regime/research, and adaptive env flags are
no longer read.
